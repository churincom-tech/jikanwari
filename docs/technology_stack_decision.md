# Technology Stack Decision

## 決定
最小試作は、完全無料のローカル静的Webアプリとして開発します。

採用する初期技術スタック:

| 項目 | 採用 |
| --- | --- |
| 実行環境 | ローカルブラウザ |
| UI | HTML |
| スタイル | CSS |
| ロジック | Vanilla JavaScript |
| データ保存 | ブラウザ内状態、JSONエクスポート/インポート、必要に応じてlocalStorage |
| 生成処理 | ブラウザ内JavaScriptで実行 |
| サーバー | なし |
| 有料API | 使用しない |
| 外部API | 使用しない |
| 外部CDN | 初期試作では使用しない |
| npm依存 | 初期試作では使用しない |
| ビルド工程 | なし |

## 理由
- 完全無料で始められる
- ユーザーのPC上だけで動く
- 教員名や学校内部情報を外部送信しない
- 開発環境の準備が軽い
- `index.html` をブラウザで開くだけで試せる
- 初期試作の入力フォーム、候補生成、候補表示には十分

## 既存プロジェクト確認
現時点で、アプリ本体の既存構成はありません。

確認した未存在項目:
- `package.json`
- `src/`
- `index.html`
- `app/`
- `public/`

そのため、既存フレームワークへの追従ではなく、軽量な静的Webアプリ構成を新規に提案します。

## 比較した候補
| 候補 | 判断 | 理由 |
| --- | --- | --- |
| 静的HTML/CSS/Vanilla JS | 採用 | 無料、ローカル、外部依存なし、最小試作に十分 |
| React + Vite | 後続候補 | UIが大きくなれば有効だが、初期はnpmとビルド工程が増える |
| Python Flask | 不採用 | サーバー起動が必要になり、利用者の準備が増える |
| Electron | 不採用 | デスクトップ配布には有効だが、初期試作には重い |
| 有料API/クラウドAI | 不採用 | ユーザー方針に反する |
| 外部最適化API | 不採用 | ローカル完結と個人情報保護に反する |

## モジュール責務
| ファイル | 役割 |
| --- | --- |
| `index.html` | 画面構造、フォーム、候補表示エリア |
| `styles.css` | レイアウト、入力フォーム、時間割グリッド、警告表示 |
| `main.js` | UIイベント、画面更新、各モジュールの接続 |
| `state.js` | 入力状態、生成結果、選択中候補の管理 |
| `sample-data.js` | 初期表示や検証用の匿名サンプルデータ |
| `timetable-core.js` | 候補生成ロジック |
| `validation.js` | 入力確認、ハード制約検証 |
| `scoring.js` | ソフト制約スコア計算 |
| `import-export.js` | JSON保存、読み込み、エクスポート |

## 生成アルゴリズムの初期方針
初期試作では、過剰な最適化ライブラリを使わず、ローカルJavaScriptで実装できる実用最小の探索にします。

1. 固定授業を先に配置する
2. 必要授業数から未配置授業リストを作る
3. 特別教室、勤務不可、担当可能枠など制約が強い授業から並べる
4. ハード制約を満たす枠へ順に配置する
5. 失敗時は候補を破棄し、並び順や探索順を変えて再試行する
6. 最大3案を生成する
7. 各候補にハード制約検証とソフト制約スコアを付ける

## ローカル保存方針
- 初期試作ではJSONエクスポートを必須にする
- localStorageは補助的な自動保存として扱う
- 個人情報を含む実データは外部送信しない
- ブラウザ外への保存はユーザー操作によるJSONダウンロードに限定する

## 後続で見直す条件
次の状態になったら、React/Viteやローカルのオープンソース最適化ライブラリ導入を検討します。

- 画面状態が複雑になり、Vanilla JSで保守しにくい
- 手修正、履歴、複数表示切替が増える
- 大規模校データで生成に時間がかかる
- より強い探索や最適化が必要になる

ただし、後続でも有料APIや外部送信は採用しません。
