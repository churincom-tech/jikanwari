# Loop Policy

## 全体ループ
State -> Plan -> Act -> Verify -> Reflect -> Update Harness -> Next Action

## コンテキスト収集ループ
- 最大2回
- 不足情報を3点以内で優先順位づけする
- 未確認事項は `context/assumptions.md` に記録する

## 設計ループ
- 最大2回
- 要件、データモデル、制約、画面、検証方法を分離する
- 重要判断はユーザー確認へ回す

## 試作ループ
- 最大3回
- サンプルデータで検証可能な範囲に限定する
- ハード制約違反があれば完成扱いしない

## 実装ループ
- 最大3回
- 既存プロジェクトがあれば構成を優先する
- 変更後は検証を行う

## レビュー・修正ループ
- 最大3回
- `review/quality_gate.md` を基準にする
- 既知失敗の再発を確認する

## ハーネス改善ループ
- 1ターン最大1回
- 作業環境に限定して改善する
- 何を変えたか、理由、防ぐ失敗を記録する

## ユーザー承認ループ
- 承認が必要な判断は `workflows/approval_flow.md` に従う
- 承認前に実装、外部送信、最終確定を進めない

## 停止条件
- ユーザー承認が必要
- 情報不足で合理的な仮定が危険
- 同じ失敗が反復上限に達した
- ハード制約違反を解消できない

## 失敗時の扱い
失敗は `loops/iteration_log.md` に記録し、再発防止が必要なら `review/known_failures.md` またはSkillへ反映します。
